## 🎯 Naming Conventions

### Files & Folders
| Type | Convention | Example |
|------|-----------|---------|
| Components | `PascalCase.tsx` | `EventCard.tsx`, `StatusBadge.tsx` |
| Pages | `camelCase.tsx` or `[slug].tsx` | `layout.tsx`, `[eventId].tsx` |
| API routes | `route.ts` in `api/[resource]/` | `src/app/api/events/route.ts` |
| Services/Lib | `camelCase.ts` | `eventService.ts`, `motoSequence.ts` |
| Custom hooks | `useXxx.ts` | `useHighVisibility.ts` |
| Tests | `*.test.ts` or `*.spec.ts` | `auth.test.ts` |

### Variables & Functions
| Type | Convention | Example |
|------|-----------|---------|
| Constants | `UPPER_SNAKE_CASE` | `SUPER_ADMIN`, `MAX_RIDERS` |
| Variables | `camelCase` | `eventId`, `riderCount`, `isActive` |
| Functions | `camelCase` with verb | `getEventById()`, `fetchRiders()`, `computeRanking()` |
| Types/Interfaces | `PascalCase` | `EventItem`, `MotoStatus`, `UserRole` |
| Booleans | `isXxx` or `hasXxx` | `isActive`, `canRegister`, `hasPermission` |
| Event handlers | `onXxx` or `handleXxx` | `onClick`, `onChange`, `handleSubmit` |

### Function Naming - Action Verbs

Follow these verb patterns:
- **`get`** - Simple data retrieval (sync or basic async)
  ```ts
  getEventById(id: string)
  getAccessibleEventIds(userId: string)
  ```
- **`fetch`** - Async data retrieval from external source
  ```ts
  fetchRiders(eventId: string)
  ```
- **`compute` / `calculate`** - Complex calculations
  ```ts
  computeQualification(riders: Rider[])
  ```
- **`build` / `create`** - Construct objects or URLs
  ```ts
  buildGoogleMapsUrl(lat: number, lng: number)
  buildCategoryBaseOrder(event: Event)
  ```
- **`normalize` / `transform`** - Data transformation
  ```ts
  normalizeAppRole(role: string)
  ```
- **`resolve`** - Complex resolution logic
  ```ts
  resolveRanking(motos: Moto[])
  ```
- **`load`** - Load configuration or data setup
  ```ts
  loadCustomSplitRules(eventId: string)
  ```
- **`list` / `find`** - Search or filter operations
  ```ts
  findActiveMoto(motos: Moto[])
  ```
- **`assert` / `validate`** - Validation/guards
  ```ts
  assertMotoEditable(moto: Moto)
  validateEventAccess(userId: string, eventId: string)
  ```

## 🎨 Component Guidelines

### Component Structure Template

```typescript
'use client'  // Add if component uses hooks/state

import React from 'react'
import Link from 'next/link'
import Image from 'next/image'
import type { SomeType } from '@/lib/types'
import { helperFunction } from '@/lib/helpers'

// Type definitions
interface ComponentProps {
  prop1: string
  prop2?: number
  onAction?: (data: string) => void
  variant?: 'primary' | 'secondary'
}

// Main component
export default function MyComponent({
  prop1,
  prop2 = 10,
  onAction,
  variant = 'primary',
}: ComponentProps) {
  // Hooks
  const [state, setState] = React.useState<string>('')

  // Effects
  React.useEffect(() => {
    // Setup
  }, [])

  // Handlers
  const handleClick = () => {
    // Logic
  }

  // Render
  return (
    <div className={`component component--${variant}`}>
      {/* JSX */}
    </div>
  )
}
```

### Component Classification

**1. Presentational Components**
- Stateless, accept props only
- No hooks (except context consumers)
- Focus on UI rendering
- Examples: `StatusBadge`, `EmptyState`, `EventCard`

**2. Container Components**
- Manage state with `useState`, `useCallback`
- Handle data fetching
- Pass data down as props
- Examples: `ThemeProvider`, `RidersClient`

**3. Layout Components**
- Used in `layout.tsx` files
- Wraps pages with common structure
- Examples: `MarketingTopbar`, `LandingTopbar`

### Component Best Practices

✅ **DO:**
- Use `export default` for single component per file
- Include prop types inline or use interface
- Use optional chaining `?.` for undefined checks
- Use semantic HTML tags (`<button>`, `<nav>`, etc.)
- Colocate related components in `src/components/`

❌ **DON'T:**
- Create subdirectories in `src/components/`
- Export multiple components from one file
- Use `any` type
- Mix business logic with UI rendering

- Handle complex algorithms

### Lib vs Services Decision Tree

```
New function needed?
├─ Is it a utility/helper? → lib/
├─ Is it a query wrapper? → lib/
├─ Is it building URLs? → lib/
├─ Is it auth/validation? → lib/
├─ Is it multi-step logic? → services/
├─ Does it compute complex results? → services/
└─ Does it modify race state? → services/
```

## 🌐 API Routes Guidelines

### Route Structure
```
src/app/api/
├── events/
│   ├── route.ts              # GET, POST events
│   └── [eventId]/
│       ├── route.ts          # GET, PUT, DELETE single event
│       └── riders/
│           └── route.ts      # Event-specific riders
├── motos/
│   └── [motoId]/
│       └── route.ts
└── auth/
    └── route.ts
```

### API Route Template

```typescript
// src/app/api/events/[eventId]/route.ts
import { NextRequest, NextResponse } from 'next/server'
import type { EventDetails } from '@/lib/eventService'
import { getEventById } from '@/lib/eventService'
import { assertEventAccess } from '@/lib/auth'

interface RouteParams {
  eventId: string
}

export async function GET(
  request: NextRequest,
  { params }: { params: Promise<RouteParams> }
) {
  try {
    const { eventId } = await params
    const token = request.headers.get('Authorization')?.replace('Bearer ', '')

    if (!token) {
      return NextResponse.json(
        { error: 'Unauthorized' },
        { status: 401 }
      )
    }

    await assertEventAccess(token, eventId)
    const event = await getEventById(eventId)

    return NextResponse.json(event, { status: 200 })
  } catch (error) {
    console.error('Error fetching event:', error)
    return NextResponse.json(
      { error: 'Failed to fetch event' },
      { status: 500 }
    )
  }
}

export async function PUT(
  request: NextRequest,
  { params }: { params: Promise<RouteParams> }
) {
  try {
    const { eventId } = await params
    const body = await request.json()

    // Validate & update
    const updated = await updateEvent(eventId, body)

    return NextResponse.json(updated, { status: 200 })
  } catch (error) {
    return NextResponse.json(
      { error: 'Failed to update' },
      { status: 400 }
    )
  }
}
```

### API Best Practices

✅ **DO:**
- Use `NextResponse.json()` for responses
- Validate request body with proper typing
- Check authorization first
- Return appropriate HTTP status codes
- Log errors for debugging
- Use type-safe params with `Promise`

❌ **DON'T:**
- Return raw objects from handlers
- Mix business logic with route handlers
- Skip authorization checks
- Return sensitive data (passwords, tokens)
- Hardcode URLs or configs

## 🔍 Type & Interface Patterns

### Type Definition Guidelines

```typescript
// lib/types.ts (or domain-specific files)

// 1. Domain model types
export interface Event {
  id: string
  name: string
  location: string
  startDate: string
  endDate: string
  status: 'planning' | 'active' | 'completed'
}

// 2. API request/response types
export interface EventCreateRequest {
  name: string
  location: string
  startDate: string
  endDate: string
}

export interface EventListResponse {
  events: Event[]
  total: number
  page: number
}

// 3. Discriminated union for results (common pattern)
export type Result<T> =
  | { success: true; data: T }
  | { success: false; error: string }

// 4. Optional fields for nullable data
export interface UserProfile {
  id: string
  name: string
  email: string
  profileImage: string | null  // Can be null
  bio?: string                 // Optional
}

// 5. Strict enums for role/status
export enum UserRole {
  SUPER_ADMIN = 'SUPER_ADMIN',
  ADMIN = 'ADMIN',
  JURY = 'JURY',
  USER = 'USER',
}
```

### Type Usage

```typescript
// ✅ DO: Use type imports
import type { Event, UserRole } from '@/lib/types'

// ❌ DON'T: Mix type and value imports
import { Event } from '@/lib/types'

// ✅ DO: Use discriminated unions for error handling
function handleResult(result: Result<Event>) {
  if (result.success) {
    console.log('Event:', result.data.name)
  } else {
    console.error('Error:', result.error)
  }
}
```

## 🔐 Authentication & Authorization

### Role Hierarchy
```typescript
// Defined in lib/roles.ts
export enum UserRole {
  SUPER_ADMIN,    // Highest privileges
  ADMIN,          // Event management
  REGISTRATION_APPROVER,  // Registration workflow
  CHECKER,        // Data verification
  JURY,           // Scoring/judging
  USER,           // Participant
  PUBLIC,         // No auth
}
```

### Authorization Patterns

```typescript
// lib/auth.ts - Query accessible resources
export async function getAccessibleEventIds(userId: string): Promise<string[]> {
  // Returns events user can access
}

// Check permission before operations
export async function assertEventAccess(userId: string, eventId: string): Promise<void> {
  const accessible = await getAccessibleEventIds(userId)
  if (!accessible.includes(eventId)) {
    throw new Error('Access denied')
  }
}

// Role-based operations
export function canApproveRegistration(role: UserRole): boolean {
  return [UserRole.ADMIN, UserRole.REGISTRATION_APPROVER].includes(role)
}
```

### Auth Headers
```typescript
// All API requests use Bearer token
const token = request.headers.get('Authorization')?.replace('Bearer ', '')
if (!token) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
```

## 🎨 Styling Standards

### Tailwind CSS Usage

**Principles:**
- Use utility classes exclusively
- No custom CSS files in components
- Semantic color names for consistency
- Responsive design with breakpoints

**Examples:**

```typescript
// ✅ DO: Use utility classes
<div className="flex items-center justify-between bg-white px-4 py-3 rounded-lg shadow-md">
  <h2 className="text-lg font-semibold text-gray-900">Title</h2>
  <button className="px-3 py-1 bg-blue-600 text-white rounded hover:bg-blue-700">
    Action
  </button>
</div>

// Component variants with conditional classes
function Button({ variant = 'primary' }: { variant: 'primary' | 'secondary' }) {
  const variantClasses = {
    primary: 'bg-blue-600 text-white hover:bg-blue-700',
    secondary: 'bg-gray-200 text-gray-900 hover:bg-gray-300',
  }
  
  return (
    <button className={`px-4 py-2 rounded ${variantClasses[variant]}`}>
      Click me
    </button>
  )
}

// ❌ DON'T: Create custom CSS files
// ❌ DON'T: Use inline styles
```

**Color Palette (via Tailwind):**
- Text: `text-gray-900`, `text-gray-600`, `text-white`
- Backgrounds: `bg-white`, `bg-gray-50`, `bg-blue-600`
- Borders: `border-gray-300`, `border-blue-200`
- Interactive: `hover:`, `focus:`, `active:` variants

## ⚠️ Error Handling

### Service Error Pattern

```typescript
// services/eventService.ts
export function processEvent(event: Event): Result<ProcessedEvent> {
  try {
    if (!event.name) {
      throw new Error('Event name is required')
    }
    
    const processed = { ...event, processed: true }
    return { success: true, data: processed }
  } catch (error) {
    const message = error instanceof Error ? error.message : 'Unknown error'
    return { success: false, error: message }
  }
}
```

### API Error Pattern

```typescript
// API route error handling
export async function POST(request: NextRequest) {
  try {
    const body = await request.json()
    const result = await processEvent(body)

    if (!result.success) {
      return NextResponse.json(
        { error: result.error },
        { status: 400 }
      )
    }

    return NextResponse.json(result.data, { status: 201 })
  } catch (error) {
    console.error('Validation error:', error)
    return NextResponse.json(
      { error: 'Invalid request body' },
      { status: 400 }
    )
  }
}
```

### Error Messages
- **User-facing errors:** Indonesian language, clear action items
- **Technical errors:** English, detailed for debugging
- **Validation errors:** Field-specific messages
- **Authorization errors:** Generic message (don't expose system details)
